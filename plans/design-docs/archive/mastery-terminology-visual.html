Item metrics

Three derived metrics per item, all 0–1:

  • Speed — how fast? (ewma → exponential decay)
  • Freshness — how recent? (half-life model)
  • Automaticity = speed × freshness

Item classification uses automaticity only:

  • Automatic: auto > 0.8
  • Working: seen, auto ≤ 0.8
  • Unseen: no data

Note the G−5 example: auto = 0.80, classified Automatic. But speed = 0.87 → heatmap-4 (yellow-green), not green. This is the tension point.

Group rollup

Level automaticity = P10 (10th percentile) of per-item automaticity across the group's items. Conservative — one weak item drags the whole group down.

Status labels (proposed)

  • Not started — no items seen
  • Hesitant — P10 auto < 0.2 (just started, still pausing)
  • Learning — P10 auto 0.2–0.8 (making real progress)
  • Automatic — P10 auto ≥ 0.8 (nearly all items fast & fresh)

Stale detection (separate)

A group can also be stale: mean(speed) ≥ 0.5 AND mean(freshness) < 0.5. This means "you were fast, but it's been a while." Used for the "Review" cue at the skill level.

Stale detection deliberately uses speed and freshness separately, not automaticity — the question is "was fast, has decayed."

Expansion gate

Level automaticity ≥ 0.7 → unlock next group.

Exhaustive, level-only status

Every level gets a status based only on its own items. No cross-level logic here (expansion gate, sequencing, etc.).

Two independent axes:

  • Speed: Automatic / Learning / Hesitant / Starting — "how fast are you at this level?"
  • Freshness: Needs review / Fresh — "have you practiced recently?"

Plus two special overrides: Skipped and Not started.

Composable statuses

A started level has both a speed status and a freshness status. e.g. "Learning + Needs review" means you weren't fast, AND it's been a while. "Automatic + Fresh" = the goal state. "Automatic + Needs review" = was fast, decayed.

Where displayed?

Currently we don't display level status directly, but we should — e.g. round-complete screen, practice tab level list. The speed status maps naturally to the heatmap legend colors.

Open: Hesitant vs Starting

Hesitant (P10 speed > 0 but < 0.3) vs Starting (P10 speed = 0) — is the distinction useful? Starting = most items unseen, a few barely tried. Hesitant = you've tried enough to have a measurable (but low) speed. Both mean "keep going." Could merge into one status.

3. Step 2: Cross-level Recommendation Algorithm

Takes level statuses + level ordering → what to practice

Inputs

  • Ordered list of levels with their status (speed + freshness)
  • Level ordering (which is "next" after the current ones)
  • Expansion gate: are you fast enough at current levels to start a new one?

Algorithm sketch

1. Identify candidates by status
Needs review Any level with freshness < 0.5 → review candidate
Learning / Hesitant Any started, non-automatic level → practice candidate
Not started Next unstarted level + expansion gate open → expand candidate
2. Prioritize
Review > Practice > Expand
(Same priority order as current system)
3. Cap
Recommend top N levels (e.g. consolidate indices + expand index)

Example: Guitar Fretboard

Level Speed Freshness Recommendation
Natural Notes Automatic Fresh — (done)
F, C Learning Fresh Practice
G♯, C♯, D♯ Learning Needs review Review
B, E, ♯♭ Not started — (gate closed)
Result: recommend Review (G♯) + Practice (F,C). Gate closed because active levels aren't all automatic.

Expansion gate

Gate opens when: all started, non-skipped levels have speed status = Automatic AND freshness = Fresh.
Expands: the next unstarted level in sequence (not all at once).

Separation of concerns

Step 1 (level status) is a pure function of that level's items. Step 2 (recommendations) uses level statuses + structure to decide what to do.

This matches the current code structure: computeRecommendations() already takes group stats and produces consolidate/expand recommendations. The change is making the intermediate level status explicit and exhaustive.

Similar to skill-level recs

The pattern is the same at both levels:

  1. Compute per-unit stats (per-level or per-skill)
  2. Classify into status
  3. Identify candidates, prioritize, cap

Expansion gate: simplified

v2 proposed "P10 speed ≥ 0.9 across all active levels." Now it's simpler: all started levels must be Automatic + Fresh. This is a higher bar than the current 0.7 automaticity threshold — you need to be truly fast AND recently practiced to unlock more.

Open: is this too strict? The current gate (P10 auto ≥ 0.7) lets you expand while still working on current levels. The new gate would require finishing each level before starting the next. Maybe the gate should be "all started levels are at least Learning" or "no Hesitant/Starting levels"?

Iteration 4 — Refined Speed Levels & Expansion Policy

Based on v3 feedback. Key changes: add "Learned" speed level (≥0.7) between Learning and Automatic. Clarify Hesitant vs Starting distinction (item coverage). Expansion gate = "all active levels at least Learned, none need review." More nuanced rec priority with expand/automate alternation.

1. Per-level Speed Axis: 5 Levels

Speed status — what's the user's experience at this level?
Status P10(speed) What it feels like
Automatic ≥ 0.9 Don't have to think about it — just know it
Learned ≥ 0.7 Remember it, but have to think for a bit
Learning ≥ 0.3 Have to think about it — working through the logic
Hesitant > 0 Seen nearly all items, but still slow on most

What P10(speed) tells us about item coverage

P10 = 10th percentile. For a 48-item level:

  • P10 = 0 (Starting): at least 5 items are unseen — still encountering new material
  • P10 > 0 (Hesitant): at most ~4 items unseen — the bar is mostly not grey, now working on speed

For a 10-item level, P10 = 0 means ≥ 1 unseen. The threshold scales naturally with level size.

Why 5 levels?

Functionally, only Automatic / Not yet matters. But finer gradations give the user a sense of progress and let us write better copy. "You've learned these — now automate them" is more motivating than "keep practicing."

Hesitant vs Starting

Both mean "early stage," but the user experience is different:

  • Starting: you're still seeing grey cells fill in. The progress feeling comes from coverage.
  • Hesitant: the bar is mostly colored now. Progress feeling shifts to speed — watching gold cells turn green.

P10 naturally captures this: it's 0 when enough items are unseen that the 10th percentile falls on an unseen item.

3. Cross-level Recommendations: Expand & Automate

Expansion gate + recommendation priority with alternation

Expansion gate

Gate opens when:
  • All started, non-skipped levels are at least Learned
  • No levels Need review
Expands: the next unstarted level in sequence (one at a time).

Recommendation priority

Priority Condition Action
1. Review Any level: Needs review Recommend reviewing that level
2. Practice Any level: Starting or Hesitant or Learning Recommend practicing that level
3. Expand Gate open + next unstarted level available Recommend starting next level
4. Automate Any level: Learned (not yet Automatic) Recommend drilling for automaticity

Expand/Automate alternation

Policy: expand if ≤ 2 "Learned" levels. If ≥ 3 "Learned" levels, prioritize automating them before expanding more.

This prevents the user from racing through all levels at "Learned" without automating any. But it also prevents the boredom of drilling one level to perfection before seeing anything new.

Pedagogically sound alternation

The expand/automate alternation prevents two failure modes:

  • Grinding one level forever (boring, diminishing returns)
  • Racing through everything at "Learned" (nothing gets automated, everything decays, massive review backlog later)

The cap (e.g. 3 Learned levels) is a tunable parameter. Start with 3, adjust based on user feedback.

Expansion gate is now natural

The gate uses the same speed thresholds as level status. "All active levels at least Learned" = P10(speed) ≥ 0.7 for each. Compare to the current system: P10(automaticity) ≥ 0.7 across all started items. Similar threshold, cleaner concept.

Future: effort-based limits

Once we track session effort (reps, time), we can add: "if you've done 150 reps on one level today, recommend something else." This slots in at the recommendation step without changing the status model.
